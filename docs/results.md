# Results

These results are from local synthetic demo data generated by the project. They are not real grid
benchmark results.

## Real Local Data

Published metrics in this repository use synthetic demo data only. For real local CSV exports,
first normalize the file into the canonical schema, then run training, backtesting, anomaly
comparison, and anomaly tuning locally.

Raw and processed data remain local under `data/raw/` and `data/processed/`, which are ignored by
Git. Real-data metrics should be generated locally and kept under ignored artifact paths.

## Rolling Backtest

Rolling-origin backtesting is more appropriate for time-series forecasting than a random split
because each fold trains on earlier observations and evaluates on later observations.

Configuration:

- Input: `data/raw/demo_energy_weather.csv`
- Model: Ridge regression
- Forecast horizon: 24
- Initial train size: 1000
- Fold size: 168
- Step size: 168
- Fold count: 5
- Random state: 42

Aggregate rolling backtest metrics:

| Metric | Value |
| --- | ---: |
| mean_mae | 46.1106 |
| mean_rmse | 65.7001 |
| mean_mape | 4.5168 |
| mean_r2 | 0.3317 |

Backtest metrics were written locally to `reports/metrics/backtest_ridge_h24.json`, which is
ignored by Git.

## Anomaly Comparison

Anomaly comparison used the same synthetic demo dataset and chronological split as the single
forecast evaluation.

Configuration:

- Model: Ridge regression
- Forecast horizon: 24
- Split method: chronological
- Test size: 0.2
- Train rows: 1574
- Test rows: 394
- Random state: 42
- Labeled anomalies in test split: 23

| Method | Configuration | Detected | Precision | Recall | Macro F1 | Confusion matrix |
| --- | --- | ---: | ---: | ---: | ---: | --- |
| `residual_zscore` | threshold `3.0` | 5 | 0.0 | 0.0 | 0.4816 | `[[366, 5], [23, 0]]` |
| `robust_residual` | threshold `3.5` | 9 | 0.0 | 0.0 | 0.4788 | `[[362, 9], [23, 0]]` |
| `isolation_forest` | contamination `auto`, 15 features | 248 | 0.0685 | 0.7391 | 0.3335 | `[[140, 231], [6, 17]]` |

The residual methods were too conservative in this split and missed all labeled anomalies.
IsolationForest achieved higher recall, finding 17 of 23 labeled anomalies, but produced many
false positives.

Anomaly comparison metrics were written locally to `reports/metrics/anomaly_comparison_h24.json`,
which is ignored by Git.

## Anomaly Tuning

Anomaly tuning used chronological train, validation, and test periods. The forecast model fits on
training rows, anomaly thresholds or contamination settings are selected on validation rows, and
the selected settings are evaluated on held-out test rows.

Configuration:

- Model: Ridge regression
- Forecast horizon: 24
- Split method: chronological
- Validation size: 0.2
- Test size: 0.2
- Train rows: 1180
- Validation rows: 394
- Test rows: 394
- Random state: 42
- Selection metric: macro F1
- Labels available: true

Selected configurations:

| Method | Selected configuration |
| --- | --- |
| `residual_zscore` | threshold `1.5`, label-based selection `true` |
| `robust_residual` | threshold `2.0`, label-based selection `true` |
| `isolation_forest` | contamination `0.01`, label-based selection `true` |

Held-out test results:

| Method | Detected | Labeled anomalies | Precision | Recall | Macro F1 | Confusion matrix |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| `residual_zscore` | 18 | 23 | 0.0 | 0.0 | 0.4726 | `[[353, 18], [23, 0]]` |
| `robust_residual` | 25 | 23 | 0.0 | 0.0 | 0.4676 | `[[346, 25], [23, 0]]` |
| `isolation_forest` | 59 | 23 | 0.0 | 0.0 | 0.4419 | `[[312, 59], [23, 0]]` |

Threshold tuning selected configurations on the validation split, but none of the tuned methods
recovered labeled anomalies in the held-out test split. This demonstrates the need for separate
validation and test periods in anomaly detection.

Anomaly tuning metrics were written locally to `reports/metrics/anomaly_tuning_h24.json`, which is
ignored by Git.
